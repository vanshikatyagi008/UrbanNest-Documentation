UrbanNest — a multi‑vendor e‑commerce platform built on .NET, Angular & PostgreSQL

UrbanNest connects independent retailers with everyday consumers on a single marketplace: retailers list and manage products, consumers browse, cart, and check out, and an admin layer keeps the marketplace healthy. This document is the single source of truth for how the system is built, how the pieces talk to each other, and why it was designed this way.

What & Why

What UrbanNest is, and why it's built this way

UrbanNest is a marketplace, not a single-shop store — that one decision shapes almost every design choice below.

1 What it is

A three-sided marketplace: Consumers shop, Retailers run their own storefronts with their own products and orders, and an Admin oversees categories, users, and moderation. Everything runs behind one Angular SPA and one .NET Web API.

2 Why this split

Splitting Users from Consumer / Retailer profile tables keeps authentication generic (one login table, one password, one role) while letting each role carry very different data — a retailer needs GST numbers and bank details; a consumer needs a date of birth and wishlist.

3 Why these technologies

.NET's Web API + EF Core gives strongly-typed, testable business logic; PostgreSQL gives relational integrity for orders and inventory (money and stock should never be "eventually consistent"); Angular's standalone components keep the frontend modular per role.

Design principle: every write that touches money or stock (placing an order, verifying a payment, updating order status) goes through a Service class that talks to the database directly inside a single request — there is no background queue, which keeps the system easy to reason about for a marketplace at this scale.
System Architecture

How the three layers fit together

A classic 3-tier layout: Angular talks only to the API over HTTPS/JSON, the API talks only to PostgreSQL through EF Core — nothing skips a layer.

Fig. 1 — Three-tier request path: Angular client → .NET Web API → PostgreSQL

Why a layered API instead of controllers talking directly to the database

Every controller (e.g. ConsumerController, RetailerController) depends only on an interface — IProduct, ICart, IOrder — never on a concrete Service class or the DataBase context directly. This is registered in Program.cs via builder.Services.AddScoped<IOrder, SOrder>() for every module. The payoff: controllers stay thin (HTTP concerns only — status codes, model binding), business rules live in one place per module, and any service can be swapped or unit-tested without spinning up the web server.

Data Layer

Database design

PostgreSQL holds 16 tables, all wired through a single EF Core DbContext called DataBase. The schema is organized around four domains that mirror the marketplace itself.

Fig. 2 — Domain map of the 16-table schema (arrows = "has / references")

Why order items carry their own status and price

An order can contain products from several different retailers. Rather than one status per Orders row, each OrderItem has its own Status (Pending → Shipped → Delivered) so one retailer can mark their line "Shipped" without affecting another retailer's line in the same checkout. Storing Price on the order item — not reading it live from Product — means a later price change never rewrites history.

Why cart/order items use NO ACTION instead of cascade delete

In DataBase.OnModelCreating, both CartItem → Product and OrderItem → Product are configured with DeleteBehavior.NoAction. If a retailer deletes a product that already has order history, that history must not vanish — cascading would silently erase paid orders. Product images, by contrast, are safe to cascade: deleting a product should also delete its images.

Platform

Authentication & security

Registration is gated behind an email OTP, login issues a signed JWT, and every subsequent request carries that token — no server-side session state.

Fig. 3 — Register-with-OTP, then token-based login
1

Register

POST /api/Auth/Register creates the account request and triggers SendOtp — no Users row exists yet until the email is proven to belong to the requester.

2

Verify OTP

EmailOTP stores the code with an ExpiryTime; VerifyOtp checks the code and expiry, then flips IsVerified and finalizes account creation. ResendOtp covers expired/lost codes.

3

Login

POST /api/Auth/Login checks credentials and returns a JWT whose claims include the user's role. Program.cs configures ValidateIssuer, ValidateAudience, ValidateLifetime, and ValidateIssuerSigningKey — all four are enforced, not just signature checking.

4

Authorize every request

The Angular HTTP interceptor attaches the token to outgoing requests; ASP.NET's JWT Bearer middleware (app.UseAuthentication() → app.UseAuthorization()) rejects anything invalid or expired before it reaches a controller.

5

Route by role on the client

Angular reads the role out of the decoded token (jwt-decode) to send Consumers to consumerNavbar/*, Retailers to retailerNavbar/*, and Admins to /admin.

Note: CORS is currently configured with AllowAnyOrigin().AllowAnyHeader().AllowAnyMethod() in Program.cs. That's convenient for development but should be tightened to the actual frontend origin(s) before a production deployment.

Platform

Core business flows

The path a rupee actually takes, from a consumer's cart to a retailer's payout status.

Fig. 4 — Checkout-to-fulfillment pipeline, with the two notification hand-offs

Cart → order, step by step

1

Cart is per-user, not per-session

Every consumer has one persistent Cart row; CartItem rows reference it plus a Product. Refreshing the browser or logging back in later restores the same cart.

2

Payment happens before the order is written

CreateOrder opens a payment order (Razorpay-style DTO), the client completes payment, and VerifyPayment checks the signature server-side — the Orders/OrderItem rows are only created once payment is confirmed, which is why PlaceOrder is a separate step from CreateOrder.

3

One order can fan out to many retailers

PlaceOrder groups cart items by RetailerId when building OrderItem rows, so a single checkout can trigger a "new order" notification to several different shops at once.

4

Status moves independently per line item

Each retailer updates only their own OrderItem.Status via UpdateOrderStatus — they never touch another retailer's line on the same order.

Platform

Notification system

A single Notification table serves both sides of the marketplace — the Type column decides who reads it and what it says.

Consumer-facing types

ORDER_PLACED, ORDER_CONFIRMED, ORDER_SHIPPED, OUT_FOR_DELIVERY, ORDER_DELIVERED, ORDER_CANCELLED, PAYMENT_SUCCESSFUL, NEW_OFFER.

Retailer-facing types

NEW_ORDER_RECEIVED, RETAILER_ORDER_CANCELLED, LOW_STOCK, PRODUCT_APPROVED, NEW_REVIEW, PAYMENT_RECEIVED.

Defining these as constants in NotificationTypes.cs instead of raw strings scattered through the codebase means INotification.SendAsync(type, ...) is always called with a known, spell-checked value — the compiler catches a typo that would otherwise silently produce a notification nobody's UI filter recognizes.

Why one table instead of two

ConsumerId and RetailerId are both nullable on Notification; exactly one is populated per row. This keeps the read/unread/mark-all-read logic in NotificationController identical for both audiences — GetMyConsumerNotifications and GetMyRetailerNotifications are the same query shape, filtered by a different foreign key.

Frontend

Angular application

One Angular 21 app (standalone components, no NgModules) with three independent "shells" hung off the same route table.

Fig. 5 — Route tree: public pages, consumer shell, retailer shell, admin

Why two "shell" routes with children instead of separate apps

Consumer pages live under consumerNavbar/* and retailer pages under retailerNavbar/* — each is a parent route whose component renders the role-specific navbar, and every child route reuses it automatically via <router-outlet>. That's why ChangePassword and address-style components only had to be built once but appear under both shells with the right chrome around them.

Cross-cutting building blocks

HTTP Interceptor

Attaches the JWT to every outgoing request from /service/*.ts so individual components never touch the token directly.

Validators

custom-validators.ts centralizes form rules (email, phone, pincode, password strength) reused across Register, Profile and Address forms.

Notification panel

A shared notification-panel component polls the unread-count endpoints and renders the bell dropdown for both Consumer and Retailer shells.

Setup & deployment

Backend

# 1. Configure the database connection Set ConnectionStrings:DefaultConnection in appsettings / secure.json Set Jwt:Issuer, Jwt:Audience, Jwt:Key # 2. Apply EF Core migrations dotnet ef database update # 3. Run the API dotnet run --project JsonWebToken # Swagger UI available in Development at /swagger

Frontend

# 1. Install dependencies npm install # 2. Point the API base URL edit src/env/environment.ts # 3. Run the dev server ng serve # http://localhost:4200 # 4. Production build ng build # outputs to dist/
Before going to production: restrict CORS to real origins, move secure.json secrets to environment variables or a secrets manager, and confirm the JWT signing key is a long, random value that's never checked into source control.
